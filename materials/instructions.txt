Привет. Как разработчик RAG-систем, я подготовил детальный, production-ready план реализации пайплайна для данной задачи. Он разбит на логические этапы, включает рекомендации по стеку, оптимизациям под соревновательный формат и стратегии минимизации галлюцинаций.

---
### 🗺️ Этап 1: Анализ данных и подготовка окружения
1. **Разбор структуры файлов**
   - `questions.csv`: `q_id`, `query` (текст вопроса).
   - `websites.csv`: предположительно `url`, `title`, `content`, `category`, `last_updated`. Необходимо подтвердить схему.
   - `sample_submission.csv`: формат `q_id,answer`. Обратите внимание на стиль ответов: часто начинается с `"Согласно предоставленным фрагментам..."` или строго `"Нет ответа"`.
2. **Очистка и нормализация**
   - Удаление HTML-тегов, markdown-артефактов, лишних пробелов.
   - Нормализация кавычек, тире, переносов строк.
   - Фильтрация пустых или слишком коротких документов (`len(text) < 50`).
3. **Выбор инфраструктуры**
   - Локальное выполнение (если `websites.csv` ~1-5 ГБ): Python 3.10+, `pandas`, `faiss-cpu`/`chromadb`, `sentence-transformers`, `openai`/`litellm` или `vLLM`.
   - Выделение ~32+ ГБ ОЗУ для индексации. Использование `mmap` для FAISS.

---
### 📦 Этап 2: Чанкинг и векторизация (Indexing)
1. **Стратегия чанкинга**
   - Рекурсивный/семантический чанкинг: `RecursiveCharacterTextSplitter` с размером `chunk_size=600-800` токенов, `overlap=100-150`.
   - Сохранение метаданных: `source_url`, `doc_title`, `chunk_index`, `category`. Это пригодится для фильтрации и трассировки.
2. **Генерация эмбеддингов**
   - Модель: `intfloat/multilingual-e5-large-instruct` или `BAAI/bge-m3` (поддерживает RU/EN, dense + sparse).
   - Батчевая обработка (`batch_size=128-256`) с прогресс-баром и сохранением checkpoint'ов.
3. **Построение индекса**
   - FAISS `IndexFlatIP` или `IndexHNSWFlat` для быстрого поиска.
   - Сохранение индекса и маппинга `ID -> текст` на диск для переиспользования.

---
### 🔍 Этап 3: Retrieval-стратегия (Поиск контекста)
1. **Гибридный поиск (Dense + Sparse)**
   - Векторный поиск: top-15 по косинусному сходству.
   - BM25/TF-IDF по ключевым словам (банковские термины, номера, названия продуктов).
   - Объединение результатов через `Reciprocal Rank Fusion (RRF)`.
2. **Реранкинг (Cross-Encoder)**
   - Модель: `cross-encoder/multilingual-MiniLM-L6-v2` или `BAAI/bge-reranker-v2-m3`.
   - Реранк топ-15 → выбор топ-3-5 наиболее релевантных чанков.
   - Кросс-энкодер значительно снижает шум и повышает точность промпта.
3. **Окно контекста**
   - Формат: `# Фрагмент 1\n[текст]\n\n# Фрагмент 2\n[текст]...`
   - Ограничение: ~4000-6000 символов, чтобы не перегружать окно LLM и не разбавлять сигнал.

---
### 🤖 Этап 4: Prompt Engineering & Генерация
1. **System Prompt (Russian)**
   ```text
   Ты — специалист службы поддержки Альфа-Банка. Отвечай только на основе предоставленных фрагментов.
   Если в контексте нет точной информации, ответь строго: "Нет ответа".
   Не выдумывай данные, не используй общие фразы, не ссылайся на внешние источники.
   Сохраняй официально-деловой стиль, краткость и точность.
   ```
2. **Few-shot примеры (из sample_submission)**
   - Включить 2-3 примера формата: вопрос → контекст → правильный ответ.
   - Особенно важно показать паттерн `"Нет ответа"` при отсутствии данных.
3. **Параметры генерации**
   - LLM: Qwen2.5-32B/72B (через vLLM/ollama), GigaChat Pro, или YandexGPT.
   - `temperature=0.1`, `top_p=0.9`, `max_tokens=512`.
   - Включить `logprobs` или `stop_sequences=["Нет ответа", "Таким образом"]` для контроля.

---
### 📊 Этап 5: Оценка, отладка и итерации
1. **Автоматическая валидация**
   - Проверка формата `q_id,answer` (UTF-8, без лишних кавычек, корректный CSV).
   - Regex-проверка на `"Нет ответа"` vs пустые строки.
   - Проверка длины ответа (отсеивание слишком длинных/коротких).
2. **LLM-as-a-Judge (опционально)**
   - Промпт для оценки: релевантность, соответствие контексту, отсутствие галлюцинаций.
   - Выборка 50-100 вопросов для ручной проверки перед финальным сабмитом.
3. **Абляционные тесты**
   - Сравнение: только dense vs hybrid vs hybrid+reranker.
   - Изменение `chunk_size`, `top_k`, промпта.
   - Логирование `q_id` с низкими скоринговыми метками для ручной донастройки.

---
### 🚀 Этап 6: Автоматизация пайплайна и формирование сабмита
1. **Архитектура скрипта**
   ```python
   load_questions() → embed_questions() → search_top_k() → rerank() → build_prompt() → llm_generate() → save_csv()
   ```
   - Использовать `asyncio` или `ThreadPoolExecutor` для параллельных запросов к LLM.
   - Кэширование ответов (`sqlite`/`jsonl`) для повторных запусков.
   - Graceful retry при 429/500 ошибках.
2. **Форматирование вывода**
   - Точное соответствие `sample_submission.csv`: `q_id,answer`
   - Экранирование запятых в ответах кавычками `"` если нужно.
   - Кодировка `utf-8-sig` для корректного открытия в Excel.
3. **Чек-лист перед загрузкой**
   - [ ] Все `q_id` из `questions.csv` присутствуют в сабмите.
   - [ ] Нет пустых ответов (кроме допустимого `"Нет ответа"`).
   - [ ] Размер файла < лимита платформы.
   - [ ] Локальная валидация через `pandas.read_csv()` без ошибок парсинга.

### 🛠️ Рекомендуемый стек (Python)
| Компонент | Библиотека/Модель |
|-----------|------------------|
| Чанкинг | `langchain.text_splitter` или `chunking-hub` |
| Эмбеддинги | `sentence-transformers` (`multilingual-e5-large`) |
| Векторный поиск | `faiss-cpu` или `chromadb` |
| Реранкер | `cross-encoder` (`bge-reranker-v2-m3`) |
| LLM API | `litellm` (единый интерфейс к GigaChat, Qwen, OpenAI) |
| Обработка CSV | `pandas`, `csv` (standard lib) |
| Логирование | `loguru` или `structlog` |

---
### ⚠️ Типичные риски и как их обойти
| Риск | Решение |
|------|---------|
| `websites.csv` не влезает в ОЗУ | Потоковая обработка чанков, индексация по частям, `mmap` для FAISS |
| LLM галлюцинирует реквизиты/ставки | Жёсткий системный промпт, `temperature=0.1`, проверка на `"Нет ответа"` |
| Низкий recall для числовых запросов (БИК, счета) | Гибридный поиск + BM25, нормализация цифр перед эмбеддингом |
| Таймауты при генерации | Батчинг, асинхронность, fallback на более лёгкую модель |
